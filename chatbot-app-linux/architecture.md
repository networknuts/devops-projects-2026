# DevOps Architecture Design Document

**Project:** AI Question Answering Web Application
**Deployment Model:** Multi-VM Linux Infrastructure

---

# 1. Purpose

This document defines the **target infrastructure architecture** for deploying the AI-powered question answering web application.

The system enables users to submit natural language questions through a web interface and receive responses generated by a large language model.

The purpose of this document is to:

* define the production deployment architecture
* describe service responsibilities
* outline network and security boundaries
* document operational considerations
* establish a scalable and maintainable system design

This architecture separates application components across multiple virtual machines to improve **security, reliability, maintainability, and scalability**.

---

# 2. System Overview

The application provides a **web-based conversational interface** where users can ask questions and receive AI-generated responses.

The application workflow is as follows:

1. A user accesses the web application in a browser.
2. The user submits a question.
3. The backend service sends the request to the OpenAI API.
4. The response from the AI model is returned to the user.
5. The question and response are stored in the database.

# 4. Architectural Objectives

The architecture is designed with the following objectives:

* isolate application layers across separate servers
* enforce security boundaries between components
* allow independent scaling of application tiers
* provide operational observability
* ensure maintainability and fault isolation
* support future infrastructure expansion

---

# 5. High-Level System Architecture

The application follows a **multi-tier architecture** consisting of the following layers:

1. Web Access Layer
2. Frontend Application Layer
3. Backend API Layer
4. Database Layer
5. Operations and Monitoring Layer

### Logical Flow

```
User Browser
     |
     v
Nginx Reverse Proxy
     |
     v

     |
     v
Backend API (FastAPI)
     |
     +-----------------> OpenAI API
     |
     v
PostgreSQL Database
```

# 6. Infrastructure Layout

The system is deployed across **five dedicated virtual machines**, each responsible for a specific layer of the architecture.

| VM   | Role                      |
| ---- | ------------------------- |
| VM-1 | Nginx Reverse Proxy       |
| VM-2 | Frontend Application      |
| VM-3 | Backend API               |
| VM-4 | PostgreSQL Database       |
| VM-5 | Monitoring and Operations |

This separation ensures clear responsibility boundaries and improves system reliability.

---

# 7. Component Architecture

## 7.1 Nginx Reverse Proxy (VM-1)

### Role

Acts as the primary entry point for all incoming web traffic.

### Responsibilities

* accept HTTP and HTTPS traffic
* serve as reverse proxy for frontend and backend services
* route requests to appropriate application components
* handle TLS termination (future enhancement)
* enforce security controls and request filtering

### Services

* nginx

### Exposed Ports

| Port | Purpose |
| ---- | ------- |
| 80   | HTTP    |
| 443  | HTTPS   |

---

## 7.2 Frontend Application Server (VM-2)

### Role

Hosts the user interface of the application.

### Responsibilities

* deliver the web-based chat interface
* capture user input
* communicate with backend API
* render responses returned by the AI model
* manage client-side application logic

### Services

* Node.js runtime
* npm package ecosystem
* optional process manager (pm2)

### Internal Port

```
3000
```

---

## 7.3 Backend API Server (VM-3)

### Role

Handles application logic and communication with external AI services.

### Responsibilities

* process incoming user prompts
* validate request payloads
* interact with OpenAI API
* process AI responses
* store conversation data in the database
* return responses to the frontend application

### Services

* Python
* FastAPI
* Gunicorn with Uvicorn workers

### Internal Port

```
8000
```

---

## 7.4 Database Server (VM-4)

### Role

Provides persistent storage for application data.

### Responsibilities

* store conversation metadata
* store chat messages
* maintain historical chat records
* support future analytics and reporting

### Database Engine

PostgreSQL

### Internal Port

```
5432
```

Database access is restricted to the backend API server.

---

## 7.5 Monitoring and Operations Server (VM-5)

### Role

Supports operational monitoring and system administration.

### Responsibilities

* centralized monitoring
* system metrics collection
* service health visibility
* operational troubleshooting
* administrative access point

### Services

* Prometheus
* Grafana
* Node Exporter
* administrative tooling

This server provides visibility into the health and performance of the infrastructure.

# 11. Security Architecture

## API Key Management

The OpenAI API key is stored only on the backend server.

Security requirements:

* never exposed in frontend code
* never stored in source control
* accessible only to backend application processes

### Recommended storage methods

* environment variables
* restricted `.env` file
* system service environment configuration

---

## Firewall Configuration

### Reverse Proxy

Allow:

* 80 (HTTP)
* 443 (HTTPS)
* SSH from trusted administration network

### Frontend Server

Allow:

* 3000 from reverse proxy
* SSH from operations server

### Backend Server

Allow:

* 8000 from frontend server
* SSH from operations server

### Database Server

Allow:

* 5432 from backend server
* SSH from operations server

### Monitoring Server

Allow:

* SSH from administration network
* monitoring traffic internally

---

## Database Security

* use dedicated application database user
* disable superuser access for application
* restrict pg_hba.conf to backend server
* enforce password authentication
* restrict database listening interface

---

## Operating System Security

Each server must implement:

* minimal installed packages
* regular OS patching
* restricted user privileges
* proper file permissions
* systemd service management

# 13. Request Processing Flow

1. A user accesses the application URL.
2. The request reaches the Nginx reverse proxy.
3. Nginx routes the request to the frontend application.
4. The frontend application loads the chat interface.
5. The user submits a prompt.
6. The frontend sends the request to the backend API.
7. The backend processes the request.
8. The backend sends the prompt to the OpenAI API.
9. The OpenAI service generates a response.
10. The backend stores the conversation in the database.
11. The response is returned to the frontend.
12. The frontend displays the AI-generated response to the user.

# 18. Failure Domains

Each infrastructure component represents a potential failure domain.

| Component         | Impact                             |
| ----------------- | ---------------------------------- |
| Reverse Proxy     | Application unavailable externally |
| Frontend Server   | User interface unavailable         |
| Backend Server    | AI processing unavailable          |
| Database Server   | Chat history storage unavailable   |
| Monitoring Server | Loss of observability              |

Isolation of components helps limit system-wide failures.

# 21. Architecture Summary

The system is deployed across five infrastructure tiers:

* **Reverse Proxy Layer** – handles public web access
* **Frontend Layer** – delivers the user interface
* **Backend Layer** – processes requests and interacts with AI services
* **Database Layer** – provides persistent data storage
* **Operations Layer** – supports monitoring and administration

This architecture provides a secure, modular, and scalable foundation for operating the AI question answering platform in a production environment.
